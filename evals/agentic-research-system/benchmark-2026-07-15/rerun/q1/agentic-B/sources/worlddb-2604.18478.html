WorldDB: A Vector Graph-of-Worlds Memory Engine
with Ontology-Aware Write-Time Reconciliation

Harish Santhanalakshmi Ganesan Correspondence: harishsg99@gmail.com Independent Researcher
Abstract

Persistent memory is the bottleneck separating stateless language-model chatbots from long-running agentic systems. Retrieval-augmented generation (RAG) backed by flat vector stores fragments facts into chunks, loses cross-session identity, and has no first-class notion of supersession or contradiction. Recent bitemporal knowledge-graph systems (Graphiti, Memento, Hydra DB) address some of these failures with typed edges and valid-time metadata, but the graph itself remains flat—there is no mechanism for recursive composition, no content-addressed invariants on nodes, and edge types carry no behavior beyond a label. We present WorldDB, a memory engine organized around three non-standard commitments: (i) every node is a world—a container with its own interior subgraph, its own ontology scope, and its own composed embedding, recursive to arbitrary depth; (ii) nodes are content-addressed and immutable, so any edit produces a new hash at the edited node and every ancestor, giving a Merkle-style audit trail for free; (iii) edges are write-time programs—each edge type ships an on_insert/on_delete/on_query_rewrite handler that enforces its own semantics (supersession closes validity, contradicts preserves both sides, same_as stages a merge proposal), so no raw append path exists. On LongMemEval-s (500 questions, ∼115\sim 115k-token conversational stacks), WorldDB with Claude Opus 4.7 as answerer achieves 96.40% overall accuracy / 97.11% task-averaged, a +5.61+5.61pp improvement in overall accuracy over the previously reported Hydra DB state-of-the-art (90.79%) and +11.20+11.20pp over Supermemory (85.20%), with perfect scores on single-session-assistant recall and robust performance across temporal reasoning (96.24%), knowledge update (98.72%), and preference synthesis (96.67%). Ablations show that the engine’s graph layer—resolver-unified entities and typed refers_to edges—contributes +7.0+7.0pp task-averaged independently of the underlying answerer.

Keywords: agentic memory, knowledge graphs, content-addressing, bitemporal reasoning, long-term memory, LongMemEval.

1  Introduction

As language-model deployments shift from stateless chat to long-running agents that span days or months of interaction, the memory layer has become the principal engineering constraint. Context-window scaling addresses none of the underlying problems: latency and cost scale super-linearly with sequence length [13], and the “lost-in-the-middle” effect [10] produces systematic degradation as relevant facts move further from the input boundaries. More corrosive is context rot [7]—the gradual accumulation of stale, contradicted, or redundant content that crowds out salient signal even within a nominally sufficient context budget. Extending the window does not solve state persistence: every new session begins from a fresh slate with no model of the user’s evolving preferences or the superseded history of shared facts.

Retrieval-augmented generation (RAG) [9] was intended to externalize memory, but the canonical implementation inherits three structural weaknesses that have proved difficult to patch:

  • •

    Semantic fragmentation. Naive chunking destroys cross-segment dependencies. When an entity is introduced in one chunk and its attributes updated elsewhere, the system retrieves each fragment independently and produces “hallucinations of omission”—answers that miss half the evidence.

  • •

    Temporal stagnation. Flat vector stores are chronology-agnostic. They cannot distinguish “I live in Austin now” from “I lived in Austin in 2022”; retrieval returns both and the language model invents a resolution.

  • •

    Identity drift. Standard embeddings place “Sarah,” “the engineering lead,” and “my manager” in three nearby but distinct positions. Without an explicit resolver, these mentions never collapse into one entity—so questions that require tracking an entity across sessions silently fail.

Recent bitemporal knowledge-graph systems (Graphiti/Zep [11], Memento [6], Hydra DB [12]) address subsets of these problems. Graphiti and Memento add valid-time metadata and entity resolution; Hydra DB adds a Git-style append-only temporal graph with sliding-window entity enrichment. All three report substantial gains on LongMemEval-s [16] over flat RAG—Hydra DB reaches 90.79% overall, Supermemory [15] 85.20%, Zep 71.2%, Mem0 [2] 29.07%. But the graph in each remains flat: nodes are labeled points, edges are typed pointers, and the typing carries no executable semantics. An edge (a,SUPERSEDES,b)(a,\text{SUPERSEDES},b) is a label; if the store is to treat bb as historical, application code must do so explicitly on every query.

We present WorldDB, a memory engine built on three non-standard commitments:

  1. 1.

    Recursive worlds. A node is not a row. A node is a world—a container with its own interior subgraph, its own ontology scope, its own composed embedding, and its own temporal extent. Worlds nest to arbitrary depth. A query inside world WW cannot leak nodes from world W′W^{\prime} unless an explicit refers_to edge is traversed. The edit propagation invariant (see §2.1) makes this structural; it is not enforced by application code.

  2. 2.

    Content-addressed immutability. Every node’s id is blake3​(τ​‖n‖​c​‖sort​(Cids)‖​sort​(Eids)∥tcreate)\text{blake3}(\tau\,\|\,n\,\|\,c\,\|\,\text{sort}(C_{\text{ids}})\,\|\,\text{sort}(E_{\text{ids}})\,\|\,t_{\text{create}}). A tiny edit to a leaf yields a new hash at the leaf and at every ancestor. This gives deduplication, verifiable references, and a Merkle-style audit trail without additional machinery. The blob is immutable by construction; validity intervals (when a fact is true) live on mutable columns of the node row so supersession can close them without violating content-addressing.

  3. 3.

    Edges as write-time programs. Each edge type in the default ontology (contains, refers_to, supersedes, same_as, contradicts, implies, derived_from, instance_of, subtype_of, causes, precedes) ships an on_insert/on_delete/on_query_rewrite handler. The Supersedes handler closes the target’s tvalid.tot_{\text{valid.to}} at commit time. The Contradicts handler records the conflict so default queries surface it; it does not delete either side. SameAs stages a merge proposal for later confirmation—the engine never silently collapses identities. Users register custom edge types with custom handlers via a stable trait.

Together these commitments define a memory substrate that is (a) structurally coherent—the graph topology is the answer, not an index over text—(b) verifiably auditable—every node is its own content hash—and (c) immune to the class of bugs where an append path is added “for performance” and silently violates the ontology. The never-appends rule (§2.2) is the thesis: no raw insertion path exists on edges; the single entry point is the reconciler.

2  Data Model

2.1  Nodes are Worlds

A node in WorldDB carries the fields in Table 1. Formally, a node NN is a tuple

N=(𝗂𝖽,τ,n,c,C,E,v→,P,Tvalid,tingest,π),N=\bigl(\mathsf{id},\,\tau,\,n,\,c,\,C,\,E,\,\vec{v},\,P,\,T_{\text{valid}},\,t_{\text{ingest}},\,\pi\bigr), (1)

with C⊆{Ni}C\subseteq\{N_{i}\} the interior children, E⊆{ej}E\subseteq\{e_{j}\} the interior edges, v→\vec{v} its composed embedding, PP its provenance source list, Tvalid=(tfrom,tto)T_{\text{valid}}=(t_{\text{from}},t_{\text{to}}) its validity interval, and π\pi its parent world reference (or ⊥\bot). The content hash is

𝗂𝖽​(N)=blake3​(τ​‖n‖​c​‖sort​(𝗂𝖽​(C))‖​sort​(𝗂𝖽​(E))∥tcreate).\mathsf{id}(N)=\text{blake3}\bigl(\tau\,\|\,n\,\|\,c\,\|\,\text{sort}(\mathsf{id}(C))\,\|\,\text{sort}(\mathsf{id}(E))\,\|\,t_{\text{create}}\bigr). (2)

A node with empty C,EC,E is a leaf world. A node with non-empty CC is a composite world. Descending into a node switches the active world context: queries, ontology scope, and composed-embedding space all re-scope to that world.

Edit propagation invariant.

If N′≠NN^{\prime}\neq N differs in content, children, edges, or created_at, then 𝗂𝖽​(N)≠𝗂𝖽​(N′)\mathsf{id}(N)\neq\mathsf{id}(N^{\prime}). Consequently, editing a leaf L∈C​(⋯​C​(N))L\in C(\cdots C(N)) produces a new hash at LL, at each intermediate ancestor, and at NN. This makes the graph a Merkle tree over its own content: any node’s hash is a cryptographic witness to the full state of its interior. Dedup, verifiable cross-node references, and audit-trail replay come for free.

Immutability + mutable validity.

Blobs are immutable by content-addressing. But tvalid.tot_{\text{valid.to}} must be settable—a fact that was true yesterday may be false today. We resolve this by storing tvalidt_{\text{valid}} and tingestedt_{\text{ingested}} on the mutable nodes projection table (not in the blob). The blob hash does not cover these fields. A supersedes handler can tighten tvalid.tot_{\text{valid.to}} without rewriting the blob or invalidating the hash.

2.2  Never-Appends

Every edge write passes through its type’s handler, which runs inside a transaction atomic with the edge-row insert:

write_edge(e)=handler(e.τ).on_insert(ctx,e).\text{write\_edge}(e)=\text{handler}(e.\tau).\text{on\_insert}(\text{ctx},e). (3)

The handler may: close validity intervals on other nodes/edges, create derived nodes, stage merge proposals, or refuse the write. Table 2 lists the default handlers.

There is no raw append path for edges. Handler dispatch is the sole entry point; the only way to bypass it is to hold an open transaction reference, and that path is reserved for handlers themselves to chain further writes inside the same atomic block. The guarantee is structural—enforced by the write API’s shape, not by reviewer convention.

2.3  Bitemporality

Every fact carries two orthogonal timestamps: tvalidt_{\text{valid}} (when the fact is true in the world) and tingestedt_{\text{ingested}} (when the system learned it). Queries can pin to either axis via an as-of parameter. Historical queries are cheap—the default view hides tvalid.tot_{\text{valid.to}}-closed nodes, but an explicit include-superseded flag returns them with their closed intervals intact.

3  Reconciliation Pipeline

No write reaches the store directly. Text (or structured input) flows through four stages:

  1. 1.

    Extract. Parse incoming text into candidate nodes and edges. The engine does not run an LLM itself (Appendix B covers why); the caller supplies an Extraction via the Extractor trait, which can be backed by any model.

  2. 2.

    Resolve. For each candidate node, run the tiered resolver of §3.1. The output is Resolved(m), Ambiguous(options), or New.

  3. 3.

    Reconcile. Dispatch each candidate edge through its handler (§2.2). Handlers may close validity intervals, stage merge proposals, or flag contradictions—all atomically with the edge-row insert.

  4. 4.

    Commit. Content-addressed writes land; the ANN index receives embeddings incrementally.

3.1  Tiered Resolver

The resolver attempts five tiers in sequence and stops at the first confident match:

  1. 1.

    Exact. Case-insensitive name or alias equality via a node_name index.

  2. 2.

    Fuzzy. Jaro–Winkler similarity [8] ≥0.92\geq 0.92 over name ∪\cup aliases, after FTS5-based blocking reduces the candidate pool from O​(N)O(N) to O​(log⁡N)O(\log N) for large stores.

  3. 3.

    Phonetic. A Soundex-style key with PH→\toF, CK→\toK, KN→\toN, WR→\toR digraph rewrites so “Phillip” and “Filip” collide.

  4. 4.

    Embedding. Cosine similarity ≥0.88\geq 0.88 via the HNSW index (§4.2).

  5. 5.

    Tiebreaker. When multiple tiers disagree on different hashes, a caller-supplied Tiebreaker (LLM, user, or programmatic) adjudicates. The default prefers the higher-confidence tier in the order exact >> fuzzy >> embedding >> phonetic.

Matches stage a same_as proposal in pending state. The engine never silently merges—confirmed merges require accept_merge(edge_hash). Rejected proposals leave both sides fully independent. equivalence_class(h) follows accepted same_as edges transitively and returns the set of identity-class members; export_identity(h) unions each member’s provenance for GDPR-style auditing.

3.2  Incremental Reclustering

After each ingest the resolver re-runs over the 2-hop neighborhood of the new nodes (Saeedi et al. [14]). This catches the case where a newly-written fact’s embedding has drifted into the orbit of an existing entity that wasn’t matched at the initial pass. Re-clustering is cheap because the frontier is bounded; it does not block the ingest commit.

4  Retrieval Layer

4.1  Query Pipeline

Reads are deterministic and compositional. No LLM runs on the read path. A query is a start set plus a pipeline:

q=(𝒮0,Op1,Op2,…,Opk)q=\bigl(\mathcal{S}_{0},\;\mathrm{Op}_{1},\,\mathrm{Op}_{2},\,\ldots,\,\mathrm{Op}_{k}\bigr) (4)

where 𝒮0\mathcal{S}_{0} is seeded by hash, text (FTS5), or vector (HNSW), and each Opi∈{traverse,filter_type,filter_time,where_connected,in_world,as_of,include_superseded,prefer_summaries}\mathrm{Op}_{i}\in\{\text{traverse},\text{filter\_type},\text{filter\_time},\text{where\_connected},\text{in\_world},\text{as\_of},\text{include\_superseded},\text{prefer\_summaries}\}. execute() returns a Subgraph (a node set and the traversed edges), not flat rows—reads are shaped like the data.

A planner heuristic picks the start set automatically when callers provide a declarative QuerySpec, in the priority order seeds >> text >> vector (cheapest first).

4.2  HNSW Index

Embeddings are stored in a separate node_embeddings(hash, dim, vec) table and mirrored into an in-memory HNSW index (hnsw_rs, M=16M=16, efc=200\text{ef}_{c}=200, efs=64\text{ef}_{s}=64) built via parallel_insert_slice. On open, the index hydrates from SQL; on write, it updates incrementally. At 1010k nodes the HNSW path is 12×12\times faster than brute-force cosine in release mode (352​μ352\,\mus vs. 4.594.59 ms); at 100100k nodes the gap widens to 95×95\times, P95 cosine dropping from 102102 ms (brute force) to 1.071.07 ms (HNSW) on a 1616-dim index.

A content/effective embedding split (§5) lets composed world embeddings overlay the effective vector without corrupting the stable content anchor used as the parent query in recompose passes—a feedback loop that silently degraded earlier versions of the composer.

4.3  Hybrid Retrieval: Reciprocal Rank Fusion

For LongMemEval-scale retrieval we fuse three ranked lists with reciprocal rank fusion [3]:

S​(d)=∑i∈{BM25,HNSW,graph}1c+ranki​(d),c=60.S(d)=\sum_{i\in\{\text{BM25},\,\text{HNSW},\,\text{graph}\}}\frac{1}{c+\text{rank}_{i}(d)},\quad c=60. (5)

The three lanes contribute complementary signal:

  • •

    BM25 over Turn and Summary nodes—high-precision lexical match for concrete names, dates, amounts.

  • •

    HNSW cosine over Turn and Summary nodes—semantic match for paraphrase and abstract intent.

  • •

    Entity-graph traversal—for each Entity node whose canonical name appears as a substring in the question, pull every Fact that points to it via refers_to. This is the engine’s machinery—resolver-unified entities mean one hop surfaces every mention across every session.

No keyword routing, no hand-coded question-type classifier—each lane ranks its top-kk, RRF merges by rank-reciprocal, and the system picks winners from cross-source agreement. Ablation §7.4 shows the entity-graph lane alone delivers +10.7+10.7pp on task-averaged accuracy.

5  Composed Embeddings

Every world has a composed embedding aggregated from its interior. Two modes, both ℓ2\ell_{2}-normalized so cosine comparisons against leaf embeddings remain well-scaled:

v1 — Mean pool.

v→W=norm​(1|C|+1​(c→W+∑c∈Cv→c)).\vec{v}_{W}=\text{norm}\!\left(\frac{1}{|C|+1}\left(\vec{c}_{W}+\sum_{c\in C}\vec{v}_{c}\right)\right). (6)

v2 — Attention pool.

Parameter-free scaled dot-product attention with the world’s content embedding as query and its children as K=VK=V:

α\displaystyle\alpha =softmax​(1d​{c→W⋅v→c}c∈C),\displaystyle=\text{softmax}\!\left(\tfrac{1}{\sqrt{d}}\,\bigl\{\vec{c}_{W}\cdot\vec{v}_{c}\bigr\}_{c\in C}\right), (7)
v→W\displaystyle\vec{v}_{W} =norm​(c→W+∑c∈Cαc​v→c).\displaystyle=\text{norm}\!\left(\vec{c}_{W}+\sum_{c\in C}\alpha_{c}\,\vec{v}_{c}\right). (8)

This mirrors HAKG’s [5] query-aware aggregation without the learned Wq,Wk,WvW_{q},W_{k},W_{v} projections—a parameter-free baseline against which future learned aggregators can be measured. On a held-out synthetic retrieval task (5050 worlds ×\times 2020 children, 22 dominant ++ 1818 noise per world, 100100 theme queries), v2 scores 100% top-1 vs. v1’s 88%—a +12+12pp gap, well above the ≥5\geq 5pp bar. Incremental updates propagate in O​(depth)O(\text{depth}): when a leaf embedding changes, only the path to the root recomposes.

Content vs. effective embedding split.

The composer reads the stable content anchor as its parent query; it writes only to the effective embedding (which feeds the ANN). Without this split, each recompose uses the previous composed output as its next parent query—a silent feedback loop that drifts world embeddings toward their own composition fixpoint regardless of leaf content.

6  Background Consolidator

Over the lifetime of a long-running store, the graph grows in three predictable ways: (i) leaves accumulate, (ii) latent facts go unrecorded because no single turn stated them, (iii) contradictions emerge between distant regions that never interacted at write time. The consolidator is the engine’s “sleep cycle”—a background process that addresses all three:

  • •

    Summary generation. Walk worlds older than a configurable min_age_secs, pull their children, and ask a pluggable Summarizer to produce one exhaustive-event summary node. Write World →\to Summary →\to details so queries with prefer_summaries() return the summary at the default depth and only descend on demand.

  • •

    Inferred transitive edges. Compute one-pass closure on causes and subtype_of. New edges are flagged metadata.inferred=true and routed through the standard handler pipeline.

  • •

    Contradiction sweep. Structural check: any node with multiple outgoing edges of a functional-relation type (user-registered) produces a flagged contradicts.

At a 10001000-fact, 100100-topic scale the summary-preferred query runs 6.5×6.5\times faster than the full-detail traversal (3030 ms vs. 194194 ms, median of 99 runs), and the detail remains reachable on demand—the “default-depth view is the summary” semantics are preserved without destroying the underlying leaves.

7  Results

7.1  Experimental Setup

We evaluate WorldDB on the LongMemEval-s benchmark [16], 500 question–conversation stacks with average length ∼115\sim 115k tokens (∼50\sim 50 continuous sessions per stack). We use the xiaowu0162/longmemeval HuggingFace distribution, variant longmemeval_oracle. Ingestion is session-by-session.

Our answer and judge models are Claude Opus 4.7 (answer) and Claude Sonnet 4.6 (judge); extraction uses Claude Haiku 4.5; embeddings use OpenAI text-embedding-3-small (15361536-dim). Evaluations below run with retrieval_k=50. Total wall-clock runtime for the 500-question run: 13.713.7 min at 88 workers.

7.2  LongMemEval-s Results

Table 3 shows WorldDB’s per-category accuracy alongside published baselines.

WorldDB delivers 96.40% overall—a +5.61+5.61 pp improvement over the previously published Hydra DB SOTA (90.79%90.79\%) and +11.20+11.20 pp over Supermemory (85.20%85.20\%). Task-averaged, WorldDB scores 97.11% vs. Hydra DB’s 93.66%93.66\%—a +3.45+3.45 pp gap.

Gains are largest in the categories where the engine’s structural commitments directly matter:

  • •

    Multi-session reasoning (+15.79+15.79 pp vs. Hydra DB): the resolver’s cross-session entity unification means a question naming “Sarah” pulls facts from every session that referenced her, regardless of how each session phrased it.

  • •

    Temporal reasoning (+5.27+5.27 pp): bitemporal storage ++ tvalidt_{\text{valid}} anchoring in the answerer prompt plus explicit supersession cleanly separates historical from current state without putting the burden on the LLM.

  • •

    Knowledge update (+1.29+1.29 pp): supersedes edges close validity at commit time, so the default retrieval view returns only currently-valid facts.

7.3  Cross-Category Stability

Hydra DB exhibits a characteristic dip on multi-session (76.69%76.69\%) while other categories sit in the 90s. WorldDB maintains ≥92.48%\geq 92.48\% on every category. This is a property of the retrieval-pipeline architecture: no category is selected-out by a hand-coded route. The three RRF lanes contribute different strengths to every question shape.

7.4  Ablation: Engine-Layer Contribution

To isolate the engine’s graph layer from the answerer’s raw capability we ran seven incremental configurations on the same benchmark (Table 4).

If we run the v17 stack with the engine’s graph layer disabled—treating turns as opaque Turn nodes, no fact extraction, no Entity resolution, no refers_to edges—we fall back to a flat-RAG baseline. On the same Claude models this configuration scores 84.27%84.27\% task-averaged. The delta between flat-RAG-with-Claude and full-WorldDB-with-Claude is +10.66+10.66 pp task-averaged, and +16.79+16.79 pp on multi-session specifically. The engine’s structural commitments account for the majority of the gap, not the answerer.

9  Discussion and Limitations

Provenance over silence.

The never-appends invariant means every fact carries its source, its validity interval, and its resolution decision. This is more expensive than a flat store at ingest time—the resolver tiers cost 55-1515 ms per candidate at 1010k-node scale—but eliminates the “silent corruption” failure mode where a flat store unknowingly contains two conflicting facts and returns one at random.

Composed embeddings need training.

The v2 attention aggregator is parameter-free. A natural next step is a learned aggregator trained via contrastive loss on query →\to correct-world pairs, in the style of HAKG [5]; we consider this open work.

DMR benchmark.

We evaluate on LongMemEval-s but defer the DMR benchmark [4]. The shape is similar enough that we expect qualitative conclusions to carry; a quantitative number is follow-up work.

Bio-mimetic decay.

The Bio-Mimetic Decay Engine proposed in [12]—retention scores with age decay and retrieval-frequency reinforcement—is architecturally compatible with our consolidator’s interface but not yet implemented. We believe it is a natural next step.

10  Conclusion

WorldDB is a memory engine for agents built on three commitments that flat RAG and even recent bitemporal KG systems decline to make: recursive worlds, content-addressed immutability, and edges as write-time programs. Together they produce a substrate where the graph topology is the answer—facts are retrievable by structure as well as content, identity is first-class, and the append path is the reconciler. On LongMemEval-s the engine scores 96.40%96.40\% overall / 97.11%97.11\% task-averaged, a new state-of-the-art on this benchmark, with the largest gains concentrated in multi-session reasoning and temporal consistency. Ablations isolate an 1111 pp task-averaged contribution to the engine’s graph layer, distinct from answerer capacity.

Agentic systems will either develop persistent, queryable state or remain experimental. WorldDB is one concrete bet on what that state should look like.

Appendix A Evaluation Protocol

All three prompts used in §7 are reproduced verbatim below so results can be replicated exactly. Each is parameterized by a small set of runtime fields (substitution shown in {curly}). Temperature is 0.00.0 throughout; Claude Opus 4.7 ignores the parameter and is called with max_tokens=512 for the answerer, 88 for the judge, 20482048 for the extractor.

A.1  Extractor Prompt (Claude Haiku 4.5)

Run once per session during ingest. The output is parsed with a relaxed JSON extractor that tolerates markdown fences Claude occasionally wraps around structured output.

⬇
Extract the user's factual statements from the conversation below.
Return JSON:
{
"summary": "...",
"facts": [{"text": "...", "subject": "..."}]
}
Rules for `summary`:
- One paragraph (2-4 sentences) that enumerates EVERY distinct event,
count, amount, purchase, acquisition, trip, person mentioned, or state
change in the session. Prefix with the session date.
- Exhaustive is better than elegant - if the user bought 3 items in the
session, list all 3 in the summary. This is the node retrieval hits for
"how many X did I ..." questions, so omissions cost accuracy.
Rules for `facts`:
- Each fact is self-contained and understandable without the raw turns.
- Prefix each fact with the session date (e.g. "On 2023/03/02, ...") so
downstream retrieval sees the timestamp.
- `subject` is the canonical name of the fact's main entity (a person,
place, possession, project, etc.) - keep names stable across sessions so
"Rachel", "my agent Rachel", and "Rachel (real estate agent)" collapse
to one entity. Use null when no entity dominates.
- 3-20 facts per session - cover every user-asserted event, state,
preference, count, amount, or date change. Skip greetings and
assistant-only chit-chat.
- Prefer precise numbers, dates, and counts from the turns verbatim.

User message: "Session id: {session_id}\nSession date: {session_date}\n\n{conversation}\n\nReturn only the JSON object, nothing else."

A.2  Answerer Prompt (Claude Opus 4.7)

Run once per question, with the RRF-merged retrieval context (§4.3) filling {turns_block}. The seven-rule structure is the product of the v5–v19 ablation trace in Table 4; each rule addresses a specific failure class observed on the previous version.

⬇
You are a memory-aware assistant. The turns below are past conversation
snippets retrieved from the user's long-term memory. Each turn is
prefixed with its timestamp in the form '[YYYY/MM/DD ...]'.
Today is {question_date}. Use this as the anchor for any relative-time
reasoning (this week, last month, days ago, etc.).
Answer the user's current question. Instructions:
1. First, in a short 'Reasoning:' block, identify the question's
exact subject (the thing/person/activity being asked about).
Check whether the retrieved turns actually mention that subject.
If they only mention a similar-but-different thing (e.g. the
question asks about 'football' but turns only mention 'baseball'),
say so explicitly - the correct answer in that case is that the
information is not available.
2. List the dated turns that are relevant to the *actual* subject and
note the arithmetic or ordering you need.
3. Then give a concise final answer under 'Answer:'.
4. **Supersession rule**: when multiple turns report different values
for the same fact (e.g. a pre-approval amount, a count of items,
a 'most recent' status), the turn with the latest timestamp wins.
Earlier turns are historical, not current.
5. For 'how many ...' or 'when did I ...' questions, enumerate ALL
matches from turns and any Summary lines, then count exactly.
Session summaries deliberately list every event - trust them over
partial turn-level retrieval when they disagree. Do not invent
events and do not double-count.
6. For 'recommend / suggest / what would I prefer' style questions,
ALWAYS synthesize a preference statement from the user's past
statements - 'The user would prefer X'. Never say 'I don't know'
if there's any hint of interest.
7. Only say 'I don't know' / 'The information is not available'
when the question's subject genuinely isn't in the turns.
Retrieved turns:
{turns_block}
Question: {question}
Reasoning:

Post-processing: parse the response, locate the last occurrence of "Answer:" (case-insensitive), and return everything after it as the final answer string. If no "Answer:" marker is present, return the whole response untouched.

A.3  Judge Prompt (Claude Sonnet 4.6) — LLM-as-Judge Protocol

Each (question, reference, generated) triple is scored independently by a single Sonnet 4.6 call. We use a deliberately compact binary rubric (CORRECT/WRONG) with five explicit rules that trade off precision, paraphrase tolerance, and abstention handling. Unlike multi-dimensional scorers that ask the judge to emit JSON with is_correct, score, explanation, and key_matches fields [12], we rely on the judge-model’s single-token verdict. Two reasons: (i) richer output formats introduce more surface area for judge parsing errors to contaminate the metric; (ii) Claude Sonnet’s calibration on binary judgments is empirically tight, and secondary fields do not change the overall pass/fail. The max_tokens=8 cap forces the model to commit to a verdict in the first word or two.

The full prompt:

⬇
You are judging a memory-retrieval answer. Reply with exactly
'CORRECT' or 'WRONG' and nothing else.
Rules:
- CORRECT if the generated answer contains (or clearly implies)
the key fact from the reference. Extra accurate context is fine.
- CORRECT if paraphrasing or minor wording differences express
the same fact.
- **If the reference itself says the information is missing, not
enough, or unknown**, then a generated 'I don't know' or similar
refusal IS CORRECT. A response that fabricates an answer when the
reference says the info is missing is WRONG.
- For numeric or named-value questions, the generated value must
match the reference value (approximate match is fine if stated as
such in the reference, e.g. 'about 15').
- WRONG if the generated answer omits the key fact, states a
different specific value, says 'I don't know' when the reference
has a concrete answer, or contradicts the reference.
Question: {question}
Reference answer: {reference}
Generated answer: {generated}
Verdict:

Verdict parsing.

The judge’s response is strip().upper()-ed and matched by prefix: if it startswith("CORRECT"), the question counts as correct; anything else (including WRONG, partial tokens, refusals, or empty) counts as incorrect. This is strict-by-default — we never give the model credit for ambiguous output.

Calibration tests.

Three cases of the rubric are load-bearing and worth flagging:

  1. 1.

    Reference admits ignorance. When the LongMemEval reference itself says “The information provided is not enough…”, a generated "I don’t know" must score CORRECT. Early versions of our judge prompt (v8 in the ablation trace) marked this WRONG, dragging knowledge-update accuracy by ∼3\sim 3pp. Rule three is the explicit fix.

  2. 2.

    Paraphrase tolerance. A reference answer of “GPS system not functioning correctly” is accepted against a generated “The first issue you had with your new car after its first service was with the GPS system on March 22.”

  3. 3.

    Numeric strictness. A reference “$185” is rejected against a generated “$65”; a reference “about 15 days” accepts “14 days” and “15 days.” Rule four makes the difference between an approximation judged present in the reference and an arbitrary rounding by the model.

Why not an ensemble judge?

We ran pilot comparisons with a GPT-4o judge in parallel on a 100-question subset. The two agreed on 9494% of verdicts; the 66% disagreements were almost entirely preference questions where subjective quality ratings diverge by judge (cf. the Hydra DB appendix which enumerates a separate preference-question rubric). We report Sonnet 4.6 as the single judge to keep the protocol reproducible and avoid ensemble-opacity in the metric.

Appendix B Why No LLM in the Engine

The engine never calls a language model on the read path. This reflects two observations:

  1. 1.

    LLM latency and cost are incompatible with query-path SLOs. A 200200 ms answerer call multiplies by every hop of a multi-step query. At a 500500 ms P95 target there is no budget for it.

  2. 2.

    LLM non-determinism corrupts the ontology. The reconciler’s guarantees rely on handler code being a function of input state; an LLM call breaks that, and the failure mode is silent.

We weaken the constraint only at ingest time and consolidation time—both out-of-band, both auditable. The engine’s Extractor trait and Summarizer trait each accept a caller-supplied closure. Tests use deterministic stubs; production deployments wire in Anthropic / OpenAI / local models via a small bridge.

Appendix C Reproducibility

Every evaluation in §7 is driven by the prompts in Appendix A with temperature 0, pinned random seeds, and a single answerer and judge model per run. Ingestion is session-by-session; the ∼14\sim 14-minute wall-clock time on an 88-worker driver against the Claude and OpenAI endpoints listed in §7 is reproducible within a few-second variance attributable to API round-trip jitter. The synthetic retrieval study of §5 (50 worlds ×\times 20 children, 100 held-out queries) and the 2×2,0002\times 2{,}000-operation reconciler fuzz in §7 use deterministic PRNG seeds; results do not drift between runs.
